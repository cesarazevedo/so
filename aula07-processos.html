⚙️ Processos e Serviços

Aula 1.7 — ps, top, kill, nice e systemctl: monitore e controle o que roda no seu servidor

🤔 O que é um Processo?

Um processo é um programa em execução. Enquanto o programa é apenas um arquivo estático armazenado no disco, o processo é a instância viva desse programa carregada na memória, com recursos alocados e em plena atividade.

Analogia: o programa é a receita de bolo escrita no papel; o processo é o bolo sendo feito no forno — dinâmico, consome energia e ocupa espaço.

Identificadores essenciais

  • PID (Process ID): número único que identifica o processo no sistema.
  • PPID (Parent PID): PID do processo que o criou (processo pai).
  • Dono: usuário que iniciou o processo; define quais recursos pode acessar.
  • Estado: situação atual — rodando, dormindo, parado ou zumbi.

Estados dos processos

🏃 Running (R)
Executando na CPU
😴 Sleeping (S)
Aguardando evento
⏸️ Stopped (T)
Pausado (Ctrl+Z)
🧟 Zombie (Z)
Encerrado, não coletado
bash — árvore de processos
# Ver a árvore de processos (quem é pai de quem) usuario@servidor:~$ pstree systemd─┬─apache2───5*[apache2] ├─sshd───sshd───bash───pstree └─mysql───mysqld

💡 Processos Zombie não devem preocupar em condições normais. São apenas registros de processos que terminaram e aguardam o pai coletar o código de saída. O sistema limpa-os automaticamente.

📋 Comando ps — Listando Processos

O comando ps (process status) exibe uma fotografia instantânea dos processos em execução. Ao contrário do top, ele não atualiza em tempo real — ideal para scripts e filtragens com grep.

bash — ps
# Todos os processos do sistema — formato Unix estendido usuario@servidor:~$ ps aux USER PID %CPU %MEM STAT COMMAND root 1 0.0 0.1 Ss /sbin/init www-data 1423 0.5 2.1 S apache2 mysql 2156 1.2 5.4 Sl mysqld # Formato System V (todas as informações) usuario@servidor:~$ ps -ef # Processos de um usuário específico usuario@servidor:~$ ps -u www-data # Buscar processo por nome com grep usuario@servidor:~$ ps aux | grep nginx www-data 1423 0.5 2.1 S nginx: worker process # Encontrar PID rapidamente pelo nome usuario@servidor:~$ pgrep apache2 1423

Colunas mais importantes do ps aux

  • USER — usuário dono do processo
  • PID — identificador único do processo
  • %CPU — porcentagem de CPU utilizada
  • %MEM — porcentagem de RAM utilizada
  • STAT — estado atual: R (rodando), S (dormindo), T (parado), Z (zumbi)
  • COMMAND — comando que originou o processo

📊 top e htop — Monitoramento em Tempo Real

Enquanto o ps tira uma foto, o top exibe um vídeo ao vivo do estado do sistema, atualizando automaticamente a cada poucos segundos. É a ferramenta padrão para diagnóstico imediato de desempenho.

top - 14:32:15 up 5 days, load average: 0.52, 0.58, 0.59 Tasks: 203 total, 1 running, 202 sleeping, 0 zombie %Cpu: 5.2 us, 2.1 sy, 92.1 id | Mem: 7976M total, 4521M used PID USER %CPU %MEM COMMAND 2156 mysql 5.3 10.2 mysqld 1423 www-data 2.1 4.1 apache2 3456 usuario 1.0 2.1 node

Atalhos interativos dentro do top

  • P — ordenar por uso de CPU (padrão)
  • M — ordenar por uso de memória
  • k — enviar sinal para matar um processo (pede o PID)
  • u — filtrar por usuário
  • q — sair do top
bash — htop
# htop: versão visual aprimorada do top (aceita mouse!) usuario@servidor:~$ sudo apt install htop usuario@servidor:~$ htop

💡 Use top -o %CPU para abrir já ordenado por consumo de CPU, ou top -d 1 para atualizar a cada 1 segundo. O htop oferece barras coloridas, suporte a mouse e navegação mais intuitiva.

💀 kill e killall — Sinais para Processos

O comando kill não serve apenas para "matar" processos — ele envia sinais que instruem o processo a executar determinada ação. Matar é apenas um dos efeitos possíveis. Entender os sinais é essencial para administrar servidores com segurança.

15 SIGTERM "Termine, por favor"
9 SIGKILL "Encerre imediatamente"
1 SIGHUP "Recarregue configuração"
bash — kill e killall
# Terminar educadamente — SIGTERM (padrão, sinal 15) usuario@servidor:~$ kill 1234 # Forçar encerramento imediato — SIGKILL (último recurso!) usuario@servidor:~$ kill -9 1234 # Recarregar configuração sem matar — SIGHUP usuario@servidor:~$ kill -1 1234 # Matar todos os processos com o nome especificado usuario@servidor:~$ killall firefox # Ver todos os sinais disponíveis usuario@servidor:~$ kill -l

⚠️ SIGKILL (kill -9) não pode ser ignorado pelo processo e encerra imediatamente sem salvar dados. Use sempre kill (SIGTERM) primeiro e recorra ao -9 apenas se o processo não responder.

⚖️ nice e renice — Prioridade de Processos

O Linux distribui o tempo de CPU entre os processos conforme sua prioridade. O comando nice define a prioridade ao iniciar um processo, enquanto renice altera a prioridade de um processo já em execução.

A escala de niceness vai de -20 (máxima prioridade) até +19 (mínima prioridade). O valor padrão é 0. Quanto mais "bonzinho" (nice) o processo, menos CPU ele exige — e por isso recebe menos.

bash — nice e renice
# Iniciar com baixa prioridade (nice +10) — não atrapalha o sistema usuario@servidor:~$ nice -n 10 ./script.sh # Iniciar com alta prioridade (só root pode usar nice negativo) root@servidor:~# nice -n -10 ./processo_critico # Mudar prioridade de um processo já rodando root@servidor:~# renice -n 5 -p 1234 1234 (process ID) old priority 0, new priority 5 # Exemplo prático: backup com prioridade reduzida usuario@servidor:~$ nice -n 19 tar -czf backup.tar.gz /home

⚠️ Apenas o root pode aumentar a prioridade (valores negativos de nice). Usuários comuns só podem diminuir a prioridade dos próprios processos (valores positivos).

🔧 systemctl — Gerenciando Serviços

Serviços (ou daemons) são processos que rodam permanentemente em background: Apache, MySQL, SSH, Nginx etc. No Linux moderno, todos são gerenciados pelo systemd por meio do comando systemctl.

💡 Diferença fundamental: start/stop controlam o serviço agora. enable/disable controlam se o serviço inicia automaticamente no boot.

systemctl start nginx

Inicia o serviço imediatamente

systemctl stop nginx

Para o serviço imediatamente

systemctl restart nginx

Para e reinicia o serviço

systemctl reload nginx

Recarrega configuração sem parar

systemctl enable ssh

Habilita início automático no boot

systemctl disable ssh

Desabilita início automático no boot

systemctl status nginx

Exibe estado atual e logs recentes

systemctl is-enabled nginx

Verifica se está habilitado no boot

bash — systemctl status
root@servidor:~# systemctl status nginx ● nginx.service - A high performance web server Loaded: loaded (/lib/systemd/system/nginx.service; enabled) Active: active (running) since Seg 2025-12-01 10:00:00 Main PID: 1423 (nginx) # Listar todos os serviços ativos root@servidor:~# systemctl list-units --type=service --state=active # Ver serviços com falha root@servidor:~# systemctl --failed

📓 journalctl — Logs do systemd

O journalctl é a ferramenta para consultar os logs centralizados gerenciados pelo systemd. Em vez de verificar dezenas de arquivos dispersos em /var/log/, você tem um único ponto de acesso com filtros poderosos.

bash — journalctl
# Logs de um serviço específico root@servidor:~# journalctl -u nginx # Monitorar logs em tempo real (equivalente ao tail -f) root@servidor:~# journalctl -u nginx -f # Últimas 50 linhas de log do sistema root@servidor:~# journalctl -n 50 # Logs desde hoje root@servidor:~# journalctl --since "today" # Apenas erros e alertas críticos root@servidor:~# journalctl -p err # Logs do boot atual root@servidor:~# journalctl -b

💡 Combine journalctl -u nginx -f com systemctl restart nginx em outra aba do terminal para ver os logs de reinicialização em tempo real — técnica essencial para diagnóstico de serviços.

Caso prático: servidor lento

bash — investigando lentidão
# 1. Verificar carga geral admin@servidor:~$ uptime load average: 8.52, 7.21, 5.43 # carga alta! # 2. Identificar o processo culpado admin@servidor:~$ top -o %CPU 9876 95.2 python loop_infinito.py # vilão! # 3. Encerrar o processo problemático admin@servidor:~$ kill 9876 # 4. Se não responder, forçar encerramento admin@servidor:~$ kill -9 9876 # 5. Confirmar melhora admin@servidor:~$ uptime load average: 0.52, 2.21, 4.43 # normalizado!

🛠️ Atividade Prática — Monitoramento do Sistema

⏱ ~30 min 💻 Ubuntu Server / Terminal 📸 Tirar screenshot da saída
1
Use o top para ver os processos em tempo real
Execute top no terminal. Observe os processos ordenados por CPU. Pressione M para ordenar por memória e P para voltar à ordenação por CPU. Tire um screenshot mostrando os 5 primeiros processos. Pressione q para sair.
2
Use o ps aux para listar todos os processos
Execute ps aux e observe a saída. Em seguida execute ps aux | wc -l para contar quantos processos estão rodando. Anote o número.
3
Identifique um processo pelo nome
Execute ps aux | grep bash para localizar o processo bash em execução. Observe o PID exibido. Confirme com pgrep bash — os números devem ser compatíveis.
4
Envie um sinal com kill (sem matar nada essencial)
Abra um segundo terminal e execute sleep 300 & para criar um processo em background. Anote o PID exibido. No primeiro terminal, execute kill PID (substituindo PID pelo número anotado). Confirme que o processo não aparece mais com ps aux | grep sleep.
5
Gerencie o serviço SSH com systemctl
Execute os seguintes comandos em sequência: sudo systemctl status ssh (observe o estado), sudo systemctl restart ssh e novamente sudo systemctl status ssh. Verifique que o serviço reiniciou com sucesso e o PID mudou.
6
Verifique logs com journalctl
Execute sudo journalctl -u ssh -n 20 para ver as últimas 20 linhas do log do serviço SSH. Você deve ver o registro do restart realizado no passo anterior. Tire um screenshot mostrando as entradas de log com o horário do restart.
📌 Para refletir: Um administrador de sistemas Linux passa boa parte do dia monitorando processos, ajustando prioridades e gerenciando serviços. Dominar ps, top, kill, systemctl e journalctl é o que separa um operador de um verdadeiro SysAdmin — você acaba de dar esse passo.
