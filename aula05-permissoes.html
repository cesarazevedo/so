Aula 1.5 — chmod, chown, chgrp e umask: quem pode fazer o quê no sistema Linux

📋 Conceito de Permissões no Linux

No Linux, todo arquivo e diretório possui um conjunto de permissões que define quem pode fazer o quê. Esse sistema é fundamental para a segurança do sistema operacional: sem ele, qualquer usuário poderia ler, modificar ou apagar arquivos de outros — inclusive arquivos do sistema.

As permissões são associadas a três categorias de usuário:

  • user (u) — o dono do arquivo
  • group (g) — o grupo ao qual o arquivo pertence
  • other (o) — todos os demais usuários do sistema

O primeiro caractere indica o tipo: - para arquivo comum, d para diretório, l para link simbólico. Os nove caracteres seguintes são três trios de rwx: um para o dono, um para o grupo e um para os demais.

bash — ls -l
usuario@servidor:~$ ls -l total 12 drwxr-xr-x 2 usuario devs 4096 jan 10 Documentos/ -rw-r--r-- 1 usuario devs 1024 jan 10 config.txt -rwxr-x--- 1 usuario devs 2048 jan 10 deploy.sh

🔒 Regra de ouro: o Linux verifica as permissões sempre nesta ordem — dono, grupo, outros. A primeira categoria que corresponde ao usuário é a que vale; as demais são ignoradas.

🔢 Notação Simbólica e Octal

As permissões podem ser representadas de duas formas: simbólica (letras rwx) e octal (números 0 a 7). Ambas descrevem o mesmo conjunto de permissões e você precisa conhecer as duas.

Valores das permissões

  • r (read / leitura) = 4
  • w (write / escrita) = 2
  • x (execute / execução) = 1
  • - (sem permissão) = 0

O valor octal de cada categoria é a soma dos valores das permissões concedidas.

Exemplos de notação completa (três dígitos)

  • 755 = rwxr-xr-x — dono: tudo; grupo e outros: ler e executar
  • 644 = rw-r--r-- — dono: ler e escrever; demais: só ler
  • 700 = rwx------ — dono: tudo; grupo e outros: nada
  • 600 = rw------- — dono: ler e escrever; demais: nada

💡 Dica prática: 755 é o padrão para scripts e programas; 644 para arquivos de texto e configuração; 600 para chaves SSH e arquivos sensíveis.

🔧 chmod — Alterando Permissões

O comando chmod (change mode) altera as permissões de arquivos e diretórios. Aceita tanto a notação octal quanto a simbólica.

Modo octal

chmod — modo octal
# chmod [permissão_octal] [arquivo] usuario@servidor:~$ chmod 755 script.sh # rwxr-xr-x — dono executa, demais só leem e executam usuario@servidor:~$ chmod 644 config.txt # rw-r--r-- — dono lê/escreve, demais só leem usuario@servidor:~$ chmod 600 ~/.ssh/id_rsa # rw------- — somente o dono lê/escreve (chave privada SSH) usuario@servidor:~$ chmod -R 755 projeto/ # -R aplica recursivamente em todo o diretório

Modo simbólico

A notação simbólica usa letras para identificar a categoria (u, g, o, a), o operador (+, -, =) e as permissões (r, w, x).

ExemploSignificado
chmod u+x script.shAdiciona execução para o dono
chmod g-w arquivo.txtRemove escrita do grupo
chmod o=r arquivo.txtDefine outros: somente leitura
chmod a+x script.shAdiciona execução para todos
chmod ug+rw doc.pdfAdiciona leitura e escrita para dono e grupo
chmod u-x,o-x binRemove execução do dono e de outros

⚠️ Nunca use chmod 777 em produção! Isso concede todas as permissões a todos os usuários — uma brecha grave de segurança. Use sempre a permissão mínima necessária.

👤 chown e chgrp — Mudando Proprietário

Além das permissões em si, cada arquivo possui um dono e um grupo. Os comandos chown e chgrp permitem alterar esses atributos. Apenas o root (ou um usuário com sudo) pode mudar o dono de um arquivo.

chown — Change Owner

chown e chgrp
# Mudar somente o dono root@servidor:~# chown joao arquivo.txt # Mudar dono e grupo ao mesmo tempo root@servidor:~# chown joao:devs projeto/ # Recursivo (toda a pasta) root@servidor:~# chown -R www-data:www-data /var/www/html # Mudar somente o grupo com chgrp root@servidor:~# chgrp vendas relatorio.pdf # Verificar o resultado usuario@servidor:~$ ls -la projeto/ drwxr-xr-x 3 joao devs 4096 jan 10 projeto/

💡 Atalho: chown usuario:grupo equivale a rodar chown usuario e chgrp grupo juntos — mais eficiente e menos propenso a erros.

★ Permissões Especiais: SUID, SGID e Sticky Bit

Além do trio rwx, o Linux possui três permissões especiais que alteram o comportamento de execução ou acesso. São representadas por um quarto dígito octal (à esquerda dos três habituais) ou por letras especiais na notação simbólica.

SUID — Set User ID (4xxx)

Quando aplicado a um executável, faz o programa rodar com as permissões do dono do arquivo, não do usuário que o executa. Exemplo clássico: /usr/bin/passwd — qualquer usuário precisa alterar sua senha, mas o arquivo /etc/shadow só pode ser editado pelo root.

permissões especiais
# Aplicar SUID (s no lugar do x do dono) root@servidor:~# chmod u+s /usr/bin/meu_prog -rwsr-xr-x 1 root root 12345 jan 10 /usr/bin/meu_prog # Aplicar SGID em diretório (arquivos herdam o grupo) root@servidor:~# chmod g+s /var/projeto drwxrwsr-x 2 root devs 4096 jan 10 /var/projeto/ # Sticky Bit em diretório (só o dono pode deletar seus arquivos) root@servidor:~# chmod +t /tmp drwxrwxrwt 20 root root 4096 jan 10 /tmp/ # Notação octal: 4=SUID, 2=SGID, 1=Sticky root@servidor:~# chmod 4755 /usr/bin/meu_prog root@servidor:~# chmod 2775 /var/projeto root@servidor:~# chmod 1777 /tmp
  • SUID (4xxx): executável roda como o dono — útil para programas que precisam de privilégio elevado por um instante
  • SGID (2xxx): em diretórios, novos arquivos herdam o grupo do diretório — ideal para pastas de projetos compartilhados
  • Sticky Bit (1xxx): em diretórios, cada usuário só pode deletar seus próprios arquivos — padrão em /tmp

⚠️ SUID em scripts Shell não tem efeito na maioria dos sistemas Linux modernos por razões de segurança. Funciona apenas em binários compilados.

🧹 umask — Permissão Padrão para Novos Arquivos

O umask (user file creation mask) define quais permissões são retiradas automaticamente ao criar novos arquivos e diretórios. Funciona como uma "máscara subtrativa": os bits ativados no umask são removidos das permissões padrão.

As permissões máximas de partida são 666 para arquivos (sem execução por padrão) e 777 para diretórios. O umask é subtraído dessas permissões base.

umask
# Ver o umask atual usuario@servidor:~$ umask 0022 # umask 022: arquivos ficam com 644, diretórios com 755 # 666 - 022 = 644 | 777 - 022 = 755 # Definir umask mais restritivo (apenas dono acessa) usuario@servidor:~$ umask 077 # arquivos ficam com 600, diretórios com 700 # Testar criação de arquivo com novo umask usuario@servidor:~$ touch teste.txt && ls -l teste.txt -rw------- 1 usuario usuario 0 jan 10 teste.txt # Para tornar o umask permanente, adicione ao ~/.bashrc # echo "umask 027" >> ~/.bashrc

🔧 O umask padrão 0022 é equilibrado: permite que outros usuários leiam seus arquivos mas não os modifiquem. Para ambientes mais seguros (servidores), use 0027 (sem acesso para "outros") ou 0077 (somente o dono).

🛡 Exemplos Práticos de Segurança

Permissões corretas são a primeira linha de defesa de um sistema Linux. Veja como aplicar o conceito de privilégio mínimo em situações reais de administração de sistemas.

Servidor web (Nginx / Apache)

permissões — servidor web
# Diretório do site: dono e grupo www-data, 750 root@servidor:~# chown -R www-data:www-data /var/www/html root@servidor:~# chmod -R 750 /var/www/html # Arquivos de configuração: apenas root lê/escreve root@servidor:~# chmod 600 /etc/nginx/nginx.conf # Scripts de deploy: apenas o dono executa root@servidor:~# chmod 700 /opt/scripts/deploy.sh # Arquivo de senhas: absolutamente privado root@servidor:~# chmod 000 segredo.txt usuario@servidor:~$ cat segredo.txt cat: segredo.txt: Permissão negada

Verificação de permissões suspeitas

auditoria de permissões
# Encontrar arquivos com SUID no sistema root@servidor:~# find / -perm -4000 -type f 2>/dev/null # Encontrar arquivos com permissão 777 (perigoso!) root@servidor:~# find /var/www -perm 777 # Ver permissões detalhadas de um arquivo usuario@servidor:~$ stat config.txt File: config.txt Access: (0644/-rw-r--r--) Uid: ( 1000/usuario) Gid: (1000/usuario)

✅ Checklist de permissões seguras: arquivos de configuração devem ser 640 ou 600; scripts executáveis 750 ou 700; diretórios públicos 755; nunca use 777 em produção.

🛠️ Atividade Prática — Laboratório de Permissões

⏱ ~30 min 💻 Ubuntu Server / Terminal 📸 Tirar screenshot da saída
1
Criar estrutura de diretórios
Crie a estrutura base para o laboratório: mkdir -p ~/lab-perm/{publico,privado,scripts}. Em seguida crie arquivos de teste: touch ~/lab-perm/publico/doc.txt ~/lab-perm/privado/senha.txt ~/lab-perm/scripts/deploy.sh. Verifique com ls -lR ~/lab-perm/.
2
Aplicar permissões adequadas
Configure as permissões corretas para cada arquivo: chmod 644 ~/lab-perm/publico/doc.txt (leitura pública), chmod 600 ~/lab-perm/privado/senha.txt (só o dono), chmod 750 ~/lab-perm/scripts/deploy.sh (script para dono e grupo).
3
Testar acesso com diferentes usuários
Tente acessar o arquivo privado como root e como usuário comum: sudo cat ~/lab-perm/privado/senha.txt deve funcionar (root ignora permissões). Crie outro usuário de teste (sudo useradd -m testuser) e tente acessar com sudo -u testuser cat ~/lab-perm/privado/senha.txt — deve retornar "Permissão negada".
4
Usar chmod simbólico e octal
Pratique as duas notações: adicione execução ao script com notação simbólica: chmod u+x ~/lab-perm/scripts/deploy.sh. Depois use octal para ajustar um diretório completo: chmod -R 755 ~/lab-perm/publico/. Verifique cada mudança com ls -l.
5
Configurar umask
Verifique o umask atual com umask. Mude para umask 077 e crie um novo arquivo: touch ~/lab-perm/privado/novo.txt && ls -l ~/lab-perm/privado/novo.txt. O arquivo deve ter permissão 600. Restaure o umask padrão com umask 022.
6
Verificar com ls -la e stat
Faça uma auditoria final do laboratório: ls -laR ~/lab-perm/ para listar tudo com permissões. Use stat ~/lab-perm/privado/senha.txt para ver detalhes completos incluindo dono, grupo e permissões em octal. Tire um screenshot da saída completa.
📌 Para refletir: Permissões não são apenas uma questão técnica — são a diferença entre um servidor seguro e um comprometido. Em ambientes de produção, cada arquivo precisa ter exatamente as permissões mínimas necessárias para funcionar. Um arquivo de configuração legível por todos pode vazar senhas de banco de dados; um script executável por todos pode ser explorado por atacantes.
